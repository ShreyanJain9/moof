# architecture audit — 2026-04-29

> **the docs are good. the code is honest about being skin-deep. the
> gap between them is closeable, but only if we (a) pick the
> moofpaint-shaped forcing function deliberately, (b) cut the spec
> down to what that forcing function actually needs, and (c) re-write
> the substrate around correctness instead of around "things that
> compile."**

## scope

this memo is for the v4-restart conversation. it covers:

1. what the docs do well, briefly;
2. holes — things the docs *don't* say that the moofpaint forcing
   function needs;
3. overcomplications — places we've spec'd more than a poc requires;
4. underspecifications — places the docs use a verb but never define
   it concretely;
5. simplifications — what to cut, defer, or collapse;
6. a proposed correctness-first plan;
7. open questions for the brainstorm round.

after this round (codex + gemini) lands, we update `roadmap.md` and
the relevant concept docs, *then* delete src/ and start over.

## 1. what the docs do well

- **the lineage is deep and honest.** `vision/lineage.md` cites every
  ancestor; the synthesis claim — that smalltalk + erlang + lisp +
  datomic can be one substrate — is plausible because the docs name
  what each tradition contributes.
- **the laws are real.** `laws/substrate-laws.md`'s 16 promises are
  the right shape: "if any of these breaks, the whole moldable claim
  breaks." these are testable invariants.
- **the docs-driven discipline is correct.** the failure mode of v3
  was code-without-docs; the failure mode of v4 right now is
  docs-without-code. the second is recoverable; the first is not.
- **moldable above the rust line is the right posture.** maru showed
  this is feasible. clojure/io/self/pharo all converge on this. moof
  doesn't have to invent it.
- **state-of-the-implementation.md is exemplary.** future-us will
  thank past-us for not lying to ourselves.

## 2. holes

### H1 — replicated vats are missing entirely

the moofpaint forcing function (alice and bob both editing the same
canvas, both seeing live updates) requires *replicated state*, not
just message-passing. the docs treat collaboration as "your friend's
vat is a far-ref; you message them" (`concepts/references.md`,
`concepts/vats.md`). that's the erlang/E pattern.

what moofpaint actually needs is the **croquet pattern**:

- there is a logical *replicated vat* — same heap, same proto, same
  bytecode running on every connected machine.
- a *reflector* (a coordination role, not a heavyweight server) takes
  user inputs from all participants, totally orders them into a
  single message stream, and broadcasts that stream back.
- every replica processes the same stream, deterministically, and
  arrives at bit-identical state.

this is fundamentally different from "alice sends a message to bob's
vat." the docs gesture at it (`vision/lineage.md` cites croquet) but
*never define replicated vats as a primitive concept*.

> **fix: introduce `replicated-vat` as a substrate concept distinct
> from "regular vat." document the determinism requirements that come
> with it. give the reflector a concrete role.**

### H2 — determinism requirements are nowhere stated

croquet-style replication only works if every operation in a turn is
deterministic given the message + heap state. that means:

- no reading `$clock.wall-time` in a turn — must use a logical clock
  derived from the totally-ordered message stream.
- no `$random` from os entropy — must use a seeded PRNG whose seed is
  part of the replicated state.
- no iteration over hashmaps in undefined order.
- no addresses-of, no pointer comparisons.
- no os-environmental reads (`$env`, `$fs` paths, …).

the docs talk about purity (`laws/purity-and-effects.md`) and
capabilities (`concepts/capabilities.md`) but *purity is not the same
as determinism*. a "pure" function calling `$clock.wall-time` is not
pure (because $clock is a cap), but a function calling
`$logical-clock.now` could be pure-and-deterministic.

> **fix: add a `laws/determinism-laws.md` document. specify the
> "deterministic capability" subset: `$logical-clock`,
> `$logical-random`, `$replicated-input`. forbid wall-clock and os-
> entropy from inside a replicated vat's turn.**

### H3 — the network/transport story is missing

`concepts/references.md` says "far-refs route via local | unix-socket
| tcp | websocket." `laws/isolation-laws.md` says "the routing table
maps vat-id to location."

but **there is no doc describing the wire format, message envelope,
flow control, reconnection behavior, or message ordering guarantees
of the actual transport.** "websocket" is named; the protocol on top
of it isn't.

for moofpaint over the wire we need to know:

- what bytes go on the wire?
- when alice's machine drops, what happens to her in-flight messages?
- does the receiver re-deliver in order? lossy or reliable?
- does the reflector survive its own restart?

> **fix: add `concepts/transport.md` (or `reference/transport-wire.md`).
> spec the envelope format, the in-order/at-least-once promise, and
> the reflector lifecycle. it can be terse.**

### H4 — the GUI substrate is undeclared

moofpaint is, after all, a paint program. there must be:

- a *canvas* primitive (pixel grid).
- an *event* primitive (mouse / keyboard / touch).
- a *render-loop* primitive.

the docs hint at "morphic" via `concepts/moldability.md` and the
manifesto. nothing concrete. `$screen` is mentioned as a cap.
`$keyboard` likewise.

> **fix: write a `concepts/canvas-and-input.md` doc. specify what
> $screen / $keyboard / $pointer caps actually do at the substrate
> level. the morphic vocabulary can come later; the i/o leaves can't.**

### H5 — the moofpaint forcing function isn't named

`docs/roadmap.md` lists phases 1–7 but no concrete user-facing demo.
the manifesto's "test" is broad. there is no "this specific app, with
this specific shape, is the proof." adding moofpaint as the *named*
forcing function for "phase 5: distribution" (or earlier) gives every
phase below it a sharper goal.

> **fix: add moofpaint as the named phase-5 forcing function. write
> a short `concepts/moofpaint.md` describing what it is and what it
> demands of the substrate.**

## 3. overcomplications

### O1 — user-defined operatives in phase 1

`concepts/sends-and-calls.md` and `syntax/binding-and-defs.md` both
make `defop` a user-facing first-class feature. but operatives have a
huge cost:

- bytecode pre-compilation can't fully resolve calls — the compiler
  has to assume any name might rebind to an operative at any time.
- inline caching is harder — operatives don't have a fixed call shape.
- macros become dynamic, which means hygiene problems.
- error messages get harder.

kernel and maru pay this cost; both are *slow*. clojure's compromise
is "macros at compile-time only" — that's what we want too, in
practice, regardless of philosophy.

> **fix: in phase 1–3, special forms are hardcoded in the rust
> compiler. `defop` is a phase-4+ feature, gated on having a working
> macro-hygiene story.**

### O2 — APL-flavored tables in phase 1

`concepts/tables.md` specifies broadcasting, rank-polymorphism, axis
ops, outer products, fancy indexing, scans, reductions. that's
numpy-grade work. for moofpaint we need:

- a positional sequence.
- a keyed map.
- iteration.
- random access by index or key.

> **fix: phase-1 tables are `Vec<Value> + HashMap<Symbol, Value>`
> with `at:`, `at:put:`, `length`, `keys`, `values`, iteration. APL
> ops are stdlib code in moof, added when a real workload demands.**

### O3 — datalog queries in phase 4

`concepts/queries.md` describes a fully-recursive datalog with
stratified negation and aggregations. implementing this correctly is
a research-grade effort (souffle is a phd-thesis-shaped project).
~800 LoC is wildly optimistic.

> **fix: phase-4 query system is *list comprehensions* + *predicate
> filters* over Form-graphs. no rules, no recursion, no negation.
> datalog is "phase 7+ — when justified by a concrete need."**

### O4 — the type system

`concepts/types.md` describes nominal + refinement + structural +
function + dependent + intersection + union types. dependent types
are an active research area; sound implementations are rare even in
agda-grade languages.

> **fix: phase-4 type system is nominal (proto-chain check) +
> structural (record shape) + intersection/union via combinators.
> refinement types are phase-6 (predicates ride alongside, but the
> analyzer doesn't *prove* them — it runtime-checks). dependent
> types are deleted from the docs until we have a clear, narrow
> use case.**

### O5 — number tower depth

`concepts/numbers.md` specifies integer (small + bigint), float,
rational, complex. four representations, with promotion rules.

for moofpaint and the next year of work: i64 + f64 are sufficient.
bigint comes when overflow is observed (which it will be, but later).
rational and complex can wait until a workload demands them.

> **fix: phase-1 numbers are i64 + f64. bigint at first overflow.
> rational/complex deleted from phase-1 docs.**

### O6 — four-faces-on-every-Form is overstated

the docs claim every Form has four faces (structure / identity /
liveness / history). but liveness only fires on vats. so on 99% of
values, it's an unused field.

a cleaner story: every Form has structure + identity + history (the
universal three). vats are Forms that *additionally* have liveness.
that's the same model with one less load-bearing claim.

> **fix: rephrase `concepts/forms.md` and `vision/manifesto.md` —
> "three universal faces; vats add a fourth." the design is unchanged;
> the prose is more honest.**

### O7 — the `become:` primitive in phase 1

become: is conceptually beautiful but operationally expensive (every
existing reference must update — usually via an indirection table or
a heap walk). it interacts hairily with inline caches, journaling,
and as-of replay.

> **fix: become: is a phase-3+ feature, gated on having (a) a stable
> id-indirection in the heap, (b) an IC-invalidation protocol, and
> (c) a journal entry shape that captures it. drop from phase 1–2.**

## 4. underspecifications

### U1 — vat representation

a vat is a Form with proto `Vat` and extra liveness face. but: where
exactly does the heap live? how is the inbox represented? what's
journaled when the vat handles a message? no doc walks through the
concrete layout.

### U2 — `become:` semantics

acknowledged in O7. compounds.

### U3 — bytecode invalidation when source changes

`laws/substrate-laws.md` L5 says "editing source invalidates bytecode
caches." what does the cache key look like? when is the
recompilation triggered (eager / lazy / on-next-call)?

### U4 — far-ref serialization across saves

`concepts/persistence.md` says "far-refs serialize as their three
fields." but cap-tokens are unforgeable; if a vat's cap-token is a
random nonce, saving and loading it preserves the bytes but not
necessarily the *trust relationship*. is the cap-token a public key?
a derived hash? something else?

### U5 — promise pipelining edge cases

`laws/isolation-laws.md` I7 says "sending to a promise pipelines."
but: what if the promise is broken? what if the same promise is
pipelined-into from two senders? what if it's pipelined-into and
then the pipeline result is awaited? the implementation has to
choose; the docs don't.

### U6 — the analyzer's bootstrap order

`concepts/types.md` says "the analyzer is moof code." but the
analyzer needs the type system to exist. and `concepts/capabilities.md`
says "the analyzer marks methods #pure" — but bootstrap stdlib code
runs *before* the analyzer is loaded. so what's the purity tag on
bootstrap code? presumably `#unknown`-then-retag, but it's never said.

### U7 — string interpolation in the bootstrap parser

the bootstrap parser is supposed to be tiny. supporting `"hi #{expr}"`
requires recursive descent with full expression parsing. so either:

- the bootstrap parser handles interpolation (and isn't tiny), or
- the bootstrap parser produces literal strings; the real parser in
  moof re-reads them (cleaner but means strings are post-processed).

### U8 — replay semantics interact with cap effects

acknowledged in `concepts/persistence.md`: "side-effects are at-most-
once; state is exactly-once." but `concepts/time-and-journal.md`
describes replay as "starting from a snapshot, replaying journal
entries." what happens to cap effects during replay? do we re-fire
`$out`'s "say" call? almost certainly no — but it's never explicit.

### U9 — the path-table

`concepts/references.md` and `concepts/image-and-world.md` mention a
path-table. it's "itself a vat." but: who handles its messages? what
caps does it need? when is a name registered? underspecified.

## 5. simplifications

cut, defer, or collapse the following for the v4-take-2 sprint.

### S1 — collapse Tables-vs-Lists for phase 1
Lists are cons-cells (necessary for code-as-data). Tables are simple
positional+keyed for phase 1; APL ops in stdlib later.

### S2 — drop user-defined operatives until phase 4
special forms hardcoded in rust; `defop` deferred.

### S3 — drop datalog until phase 6+
list comprehensions + filters are sufficient through moofpaint.

### S4 — drop dependent + refinement types until phase 6+
nominal + structural is enough.

### S5 — drop rationals/complex from phase 1
i64 + f64 only.

### S6 — drop become: from phase 1–2
phase 3+ with proper id-indirection.

### S7 — drop multi-clause pattern-matched defs from phase 1
single clause + explicit `(if …)` is enough to bring the substrate
up. multi-clause is a phase-2 ergonomic add.

### S8 — drop the inspector as morphic-canvas-glue from phase 1–3
phase 1–3 inspector is text-rendered, line-based. moofpaint is the
first canvas-rendered demo.

### S9 — drop NaN-boxing as a phase-1 concern
honest tagged enum first; nan-box when measurements show value.

### S10 — drop bytecode invalidation on source-edit until proto-mut works
phase 1–2 are immutable-after-load; live editing is phase 3.

### S11 — collapse "four faces" to "three faces + vat-extra"
docs cleanup.

### S12 — pin the world to *one* process for phase 1–4
in-process scheduler with multiple vats. cross-process is phase 5
(at the moofpaint demo).

## 6. proposed correctness-first plan

> **forcing function: `moofpaint`.** alice runs `moof moofpaint`,
> draws a stroke. bob runs `moof moofpaint join localhost:7878` in
> a second terminal, sees alice's stroke arrive in real time, draws
> her own. close both. reopen — both see the canvas. no
> reconciliation, no merge conflicts: they ran the same totally-
> ordered input stream and got bit-identical state.

if this works, moof is real. if not, we know which substrate piece
broke and we fix it.

phases:

### phase A — substrate seed (rewrite of phase 1)
forcing function: `moof '(+ 1 2)' → 3`, but *honest* this time.

deliverables:
- one Form heap kind. *every* value is a Form, even small ints
  (NaN-box later; for now, indirect through heap).
- proto-chain dispatch. inline caches at every send-site.
- `Object` proto with the reflection methods installed:
  `:proto :slots :handlers :meta :source :identity :=`.
- methods are Forms (closures). chunks are Forms. handlers are slots
  of those Forms.
- bytecode interpreter. ~30 ops.
- compiler in rust (bootstrap parser + bootstrap compiler). lib/
  bootstrap.moof loaded as source.
- test for every L1–L6 promise.

ratio: ~2500 lines of rust + ~500 lines of moof.

### phase B — vats and persistence
forcing function: a vat saves to disk per turn; quit and reboot
restores its state. `(println "hi")` works through a `$out` cap that
survives reboot.

deliverables:
- per-vat heap (every vat owns its FormIds; no cross-vat raw
  pointers).
- vat scheduler with mailbox.
- per-vat lmdb store + journal (canonical encoding).
- canonical encoding spec for Forms (binary).
- bootloader: mmap store, replay journal-tail.
- `$out`, `$clock` (logical), `$random` (seeded) caps.

ratio: ~2000 lines of rust + ~500 lines of moof.

### phase C — moldability foundations
forcing function: edit a method in moof, see the next call use the
new code.

deliverables:
- proto-handler mutation + IC invalidation.
- `does-not-understand:` extension hook.
- `become:` (with proper id-indirection).
- multi-clause pattern-matched defs.
- text-line inspector (`(inspect obj) → string`).

ratio: ~500 lines of rust + ~1500 lines of moof.

### phase D — replication and distribution
forcing function: two replicas of one vat; user inputs from each are
totally ordered by a reflector; both replicas converge.

deliverables:
- `replicated-vat` as a kind of vat (with deterministic-cap
  restrictions).
- the reflector, in moof, running in a coordinator process.
- transport: in-process channels first, then unix-socket, then
  websocket.
- envelope format spec.

ratio: ~1000 lines of rust (transports) + ~2000 lines of moof
(reflector, replica protocol).

### phase E — moofpaint
forcing function: the end-to-end test described above.

deliverables:
- `$canvas` cap (a fixed-size 1-bit pixmap, render-to-terminal first
  via half-blocks, then to web canvas).
- `$pointer` cap (mouse / touch events as a data source).
- pencil, eraser, line, rect, oval tools (in moof, as protos).
- a moofpaint vat (replicated) that holds the canvas.
- a tiny terminal UI (or web ui — TBD in phase D).
- multi-user smoke test.

ratio: ~500 lines of rust (canvas + pointer leaves) + ~2000 lines
of moof (tools, ui).

### phase F+ — everything else
real type system, datalog, multi-process, federation, package system,
APL tables, refinement types, hyperCard accessibility, web canvas,
all of it.

## 7. open questions for the brainstorm

these are the load-bearing decisions to validate against codex +
gemini:

### Q-A — replicated-vat vs message-passing as the *default* model
should `moofpaint` be a replicated vat (croquet-style; my proposal),
or a message-passing app where alice's vat far-refs into bob's vat?

argument for replicated: lower latency for shared editing; bit-
identical state; matches the user's stated "akin to croquet" goal.

argument against replicated: more substrate complexity; deterministic-
cap discipline is a lot; one-source-of-truth is simpler.

### Q-B — reflector trust model
is the reflector a *server* (centralized; one trusted machine) or a
*peer-distributed protocol* (every replica votes)?

trade-offs: server-style is simple; peer-distributed is byzantine.
for poc, server-style. but does the design *force* server-style
forever, or can we promote to peer-distributed later?

### Q-C — terminal-first vs web-first UI
moofpaint can render to terminal (half-blocks for 1-bit pixels) or
to a web canvas (websocket between server and browser). which gives
the fastest path to a working demo?

terminal is simpler (no html/js); web is more accessible to
demonstrate.

### Q-D — should the existing src/ be deleted, or refactored?
the user said "delete." the existing code is ~1700 LoC of rust + the
bootstrap.moof. we can *almost certainly* salvage parts:
- the reader (~300 LoC) is mostly correct.
- the form heap (~50 LoC) is fine.
- the bytecode interpreter (~400 LoC) needs surgery but isn't
  fundamentally wrong.

argument for delete: a clean room for the new model is easier to
align with the laws than incremental refactor.

argument against: ~1000 LoC of working code thrown away.

(my read: the user said delete; the rules are big enough that
incremental refactor would compound the dishonesty. delete.)

### Q-E — keep bootstrap.moof as-is or rewrite from scratch?
the bootstrap stdlib (~270 LoC of moof) is mostly fine — `length`,
`map`, `reduce`, etc. it's plain `(fn …)` style. we can keep it.

argument for keeping: avoid re-doing easy work.
argument for rewriting: the new substrate's calling convention may
differ (proto-Sends vs fn-calls); cleaner to redo.

(my read: keep as a reference; rewrite line-by-line into the new
substrate's idioms as those idioms stabilize.)

### Q-F — operative/applicative from day one, or never?
shutt's kernel discipline is beautiful but *expensive*. clojure's
compromise (macros at compile-time, but not first-class fexprs) is
cheaper.

i propose phase 1–3 hardcode special forms in rust; phase 4+ adds
`defop`. the brainstorm should challenge whether moof loses something
critical by pushing this to phase 4.

### Q-G — what's the single hardest thing in this plan?
my guess: making `$logical-clock` and replicated-vat play nicely
with persistence. the journal must be deterministically-replayable
*and* the wall-clock-time-of-replay must not affect state. there are
landmines in the interaction.

## post-brainstorm synthesis (2026-04-29 evening)

ran codex + gemini in parallel against this memo + the docs. both
independently arrived at *the same* unconventional answer to the
hardest problem, which is the strongest signal we get from
adversarial review. integrating now.

### convergent findings (3-of-3 or 2-of-3 across me / codex / gemini)

**S-1 — replay is three different things, conflated.**
> codex (C1) + gemini (G2) + me (n/a)
> *codex*: docs already define three things — journal-as-mutation
> log ([persistence.md](concepts/persistence.md):49), replay-as-
> message-redispatch ([vats.md](concepts/vats.md):117), and at-most-
> once cap effects ([persistence.md](concepts/persistence.md):61) —
> and conflating them means replay/undo/replication will fight each
> other. need separate log classes.
> *gemini*: at-most-once isn't enough; replay must reproduce the
> *result* of effects, not just skip them. otherwise code that
> branches on `if [$fs write: ...]` forks during replay.
> *resolution*: two journals, not one. an *input log* (totally
> ordered turn-envelopes — the replicated authoritative truth) and
> a *mutation log* (the per-replica derived journal of slot-changes,
> for snapshotting and time-travel). cap effects produce *receipts*
> that get written back into the input log as data.

**S-2 — effects-as-data (intent + receipt) is the model.**
> codex's "two-stream" + gemini's "effect-log-as-state" arrived
> independently at the same idea. accepting it.
> *the model*: a replicated vat is a pure deterministic state machine
> over ordered inputs. when its code calls `[$out say: "hi"]`, the vat
> doesn't *do* anything — it appends an `EffectIntent(turn-id,
> ordinal, payload)` to its outbox slot. a separate **authority**
> (in moof: a non-replicated effect-worker vat; in rust: a
> trampoline) reads outbox entries, executes them at-most-once,
> writes `EffectReceipt(turn-id, ordinal, result|failure)` back into
> the *input log* as a new turn. replicas never replay raw effects;
> they replay intents and receipts as ordinary deterministic data.
> *what this gets us*:
> - croquet-style convergence for state — state is a fold over the
>   input log.
> - persistence by snapshot + input-log replay (no separate WAL of
>   effects).
> - exactly-once effects via stable `(turn-id, ordinal)` ids.
> - clean trust boundary: the reflector orders authenticated user
>   inputs and effect-receipts, never ambient capability traffic.
> *cost*: one more concept (effect-worker) and one more journal
> shape (receipts as inbox-messages). cheap.

**S-3 — logical time is part of the turn envelope, not a cap.**
> codex (C4) + gemini (G3). they're right; i was wrong.
> i proposed `$logical-clock` as a deterministic-cap. but if code can
> "call time" mid-turn, you've mixed effect discipline with
> determinism discipline. instead: every turn-envelope carries a
> `(epoch, turn-seq, logical-now)` triple. inside the turn, code
> reads `[turn now]`. the substrate refuses any code in a replicated
> vat from holding a `$clock` cap.
> *resolution*: delete `$logical-clock` from the cap list. the turn
> envelope shape becomes `(session-id, epoch, turn-seq, author,
> logical-now, input-event, seed)`. seed is also envelope-bound.

**S-4 — determinism is broader than rng/clock.**
> codex (C2): the memo named clocks and rng but missed deterministic
> *allocation order, promise ids, proto-edit sequencing, hash/map
> iteration order, GC timing*. for two replicas to be bit-identical,
> all of these must be deterministic.
> *resolution*: a new `laws/determinism-laws.md` document spelling
> out *all* the substrate's determinism promises, not just the
> obvious ones. the implementation must:
> - allocate FormIds in a deterministic order keyed by turn position
>   (no per-replica clock-driven counters).
> - iterate hashmaps in insertion-order or sorted-key order, not
>   hash-bucket order.
> - never observe wall-clock or process-id during a turn.
> - run GC at deterministic checkpoints (turn boundaries), not
>   timer-driven.

**S-5 — the reflector orders inputs only, not cap-bearing app messages.**
> codex (C3) + gemini (G1).
> a reflector that sits in the path of all cross-vat traffic
> *implicitly grants* itself ambient authority over capabilities.
> that breaks `isolation-laws.md` I5 (caps ride on far-refs;
> unforgeable). 
> *resolution*: the reflector orders only authenticated *user-input
> events* (mouse/keyboard/etc) and *effect-receipts*. cross-vat
> far-ref traffic continues to flow over the regular transport
> (point-to-point), not through the reflector.

**S-6 — narrow moofpaint to "replicated stroke log + deterministic renderer."**
> codex (C5) + me (M1).
> a 1-bit pixel canvas with set-pixel ops can converge via plain
> last-write-wins (commutative). that's not what we're trying to
> prove.
> *resolution*: the moofpaint forcing function is narrower:
> - the *replicated state* is a list of `Stroke` Forms (each with an
>   author, logical-now, tool-proto, points, attrs), plus a small
>   set of *non-commutative* operations: per-user undo stack, layer
>   re-order, flood-fill (which depends on the canvas snapshot at
>   logical-now), shared selection.
> - the canvas pixmap is a *derived* view, computed by a pure
>   reducer over the stroke log.
> - rendering and pointer capture are non-replicated edge adapters.
> this design forces every hard substrate question (deterministic
> reduce, replicated proto edits, intent/receipt for save-to-disk,
> reflector ordering) without burying them under "many byte updates."

**S-7 — the substrate test gate is "two in-process replicas converge."**
> gemini's "two-headed vat" (G4) + me (M4).
> phase D's forcing function: a single rust process holds *two
> independent vat instances*, each with its own heap. an in-process
> reflector feeds both the same input log. after every turn, the
> substrate asserts `vat_a.canonical_hash() == vat_b.canonical_hash()`.
> if they ever diverge, the substrate is lying about determinism.
> this single test catches: hash-iteration nondeterminism, alloc-
> order drift, promise-id drift, GC timing leaks, etc. it's the
> tightest possible determinism harness short of a real network.

### unique-to-me findings (1-of-3) i'm keeping

these neither codex nor gemini raised, but i think they hold up:
- **M3** — phase A keeps tagged-immediates; "everything heap-allocated"
  was a perf disaster. tagged-immediates with reflection is fine;
  reflection works *through* the immediate's implicit proto.
- **M5** — proto-edits-during-replication are the *hardest* live-
  editing case; every replica must see the edit at the same logical-
  now. resolution: proto edits are turn-envelopes too — they're
  totally ordered like any other input.
- **M6** — caps don't replicate. the replicated state is forms +
  slots + handlers + meta; caps are per-replica ambient bindings
  provided by each replica's local supervisor. captured caps in
  closures are a footgun; the substrate refuses to serialize a
  closure with captured caps in a replicated vat.
- **M7** — bytecode is per-replica derived from per-replica recompile;
  only source crosses replication boundaries.
- **M8** — reflector ticks (heartbeats every N ms; batches inputs
  into ordered bundles). replicas advance logical-now by the tick
  delta. without ticks, "logical time" has no anchor between turns.
- **M9** — keep the existing src/reader.rs and src/sym.rs. throw
  away the rest.
- **M10/M11** — every phase's deliverable list adds a *test matrix*
  and a *perf budget*.

### finding i'm dropping (codex/gemini both negged)

- **memo's Q-A** — "should the *default* be replicated, or message-
  passing?" both reviews implicitly answered: keep message-passing
  as the default; replication is opt-in via supervisor at vat birth.
  i'm dropping the question.
- **memo's "logical-clock as cap"** — superseded by S-3.

### updated forcing-function for moofpaint

the demo:

> alice runs `moof moofpaint`. she draws three strokes, flood-fills
> a region, undoes one stroke. bob runs `moof moofpaint join
> wss://localhost:7878` in another window. he sees alice's three
> strokes-minus-one-undo plus the flood-fill, in the canonical
> order. he draws his own stroke, calls undo on it. alice sees bob's
> stroke arrive and disappear. each replica computes the canvas
> pixmap as a pure reducer over the shared stroke log. close both.
> reopen — both wake with the same canvas, same undo stack per user.
> the input log + snapshot is the persistent state.

### updated phase plan

(replaces section 6 above; this is what `docs/roadmap.md` will say.)

#### phase A — substrate seed (rewrite)
*forcing function*: `moof '(+ 1 2)'` → `3`, with **every** law in
`substrate-laws.md` either honored or doc'd as deferred.

deliverables:
- one Form heap kind. tagged-immediate optimization for nil/bool/
  small-int/symbol; *reflection still works* through the implicit
  proto. methods are Forms. chunks are Forms.
- proto-chain dispatch with inline caches.
- `Object` proto with the full reflection method set.
- bytecode interpreter (~30 ops).
- bootstrap reader/compiler in rust (carry over `src/reader.rs` and
  `src/sym.rs`; rewrite the rest).
- bootstrap stdlib in moof (carry over `lib/bootstrap.moof`'s
  list ops; redo the calling-convention surface).
- *test matrix*: one test per substrate law (L1–L6); one test per
  reflection-contract method.

#### phase B — single-vat persistence
*forcing function*: a vat saves per turn; quit, reboot, state
restored. `(println "hi")` works through `$out` cap that survives
reboot.

deliverables:
- per-vat heap (vat-local FormIds; no cross-vat raw pointers).
- vat scheduler (single vat for now).
- per-vat lmdb store + journal (binary canonical encoding).
- bootloader: mmap, replay journal-tail.
- the **intent/receipt model** for cap effects (S-2). caps emit
  `EffectIntent`s; a rust effect-worker executes; receipts come
  back as inbox messages.
- $out, $clock (wall, OS-bound), $random (OS-bound) caps.
  *replicated vats won't be allowed to hold these.*
- *test matrix*: snapshot + replay round-trip; effect-intent
  ordinals stable; multiple-restart idempotency.

#### phase C — moldability foundations
*forcing function*: live-edit a method via `(set-handler! Counter
:incr ...)`; next call uses the new code. inline caches invalidate.

deliverables:
- proto-handler mutation + IC invalidation.
- `does-not-understand:` extension hook.
- multi-clause pattern-matched defs (the substrate's biggest
  ergonomic win).
- text-line inspector `[obj inspect]`.
- *deferred*: `become:` (until phase D's id-indirection is in).

#### phase D — replicated vats (in-process)
*forcing function*: a single rust process runs **two replica vats**
of one logical vat. an in-process reflector feeds both the same
totally-ordered input log. after every turn, the substrate asserts
`canonical_hash(replica_a) == canonical_hash(replica_b)`.

deliverables:
- `replicated-vat` vat-mode (born this way; can't be promoted).
- the determinism laws in rust (S-4): deterministic alloc order,
  ordered hashmap iteration, no wall-clock during a turn, GC at
  turn boundaries.
- the turn envelope `(session-id, epoch, turn-seq, author,
  logical-now, input-event, seed)` (S-3).
- a tiny in-rust reflector (orders user-inputs and effect-receipts
  only; no cap traffic).
- the canonical hash function over a vat's heap.
- *test matrix*: 10k random inputs, two replicas, hash-equal at every
  turn. fault injection (drop one replica; rejoin from snapshot;
  catch up via input log).

#### phase E — moofpaint over loopback
*forcing function*: described above. one process runs the canvas
vat (replicated, two replicas), a render vat (per-replica edge
adapter), and an input-source vat (per-replica edge adapter).
inputs go to the reflector → broadcast to both replicas → both
render the same canvas.

deliverables:
- `Stroke`, `Tool`, `Layer`, `Selection` protos (in moof).
- pencil, eraser, line, rect, oval, flood-fill, lasso tools.
- per-user undo (stroke log filters by author + a logical-undo
  index).
- 1-bit pixmap data structure (rust-backed via .mco) with
  `:get-pixel:y:`, `:set-pixel:y:put:`, `:flood-fill-from:y:`.
- `$canvas` cap (renders pixmap to terminal half-blocks for first
  cut; web canvas later).
- `$pointer` cap (mouse/touch event source as DataSource).
- a moofpaint vat in moof that owns the stroke log + selection +
  layer state.
- *test matrix*: scripted input replay produces identical pixmaps
  on both replicas; persist + reload preserves canvas.

#### phase F — cross-process moofpaint
*forcing function*: alice and bob in two terminals. bob joins via
`wss://`. they edit simultaneously.

deliverables:
- websocket transport for the input log.
- reconnect-with-epoch (a rejoining replica catches up by snapshot
  + recent input log).
- *test matrix*: bob disconnects mid-stroke, reconnects, converges.

#### phase G+ — everything else
real type system, datalog, federation, package system, APL tables,
refinement types, hyperCard accessibility, web canvas, all of it.
